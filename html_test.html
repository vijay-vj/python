Quick Start

\n
\n
\n
\n
\n 
\n            from wsgiref.simple_server import make_server\nfrom pyramid.config import Configurator\nfrom pyramid.response import Response\n\ndef hello_world(request):\n    return Response(\'Hello World!\')\n\nif __name__ == \'__main__\':\n    with Configurator() as config:\n        config.add_route(\'hello\', \'/\')\n        config.add_view(hello_world, route_name=\'hello\')\n        app = config.make_wsgi_app()\n    server = make_server(\'0.0.0.0\', 6543, app)\n    server.serve_forever()\n
\n
\n
\n

Pyramid makes it easy to write web applications. You can\n start small with this "hello world" minimal\n request/response web app. This may take you far, especially while\n learning. As your application grows, Pyramid offers many features\n that make writing complex software take less effort.\n

\n

Pyramid works in all supported versions of Python. Our\n installation instructions\n will help you get Pyramid up and running.\n

\n

Pyramid\'s\n quick tutorial\n will take you step by step through writing a single file\n application, forms, database integration, and authentication.\n

\n

Developers may dive in to Pyramid\'s\n narrative documentation,\n or browse the extensive\n API reference.\n Pyramid has a rich pool of helpful resources\n from which to draw. Extending Pyramid\n is a curated and filterable list of add-ons, packages, and applications\n built to work with Pyramid.\n

\n When You Need Pyramid\n

\n
\n
\n

Megaframeworks make decisions for you. But if you don\'t fit their viewpoint, you end up fighting their decisions. Microframeworks force no decisions, making it easy to start. But as your application grows, you\'re on your own.

\n

In both cases, the focus is on the start: either too much or too little. Either way, finishing and staying finished is hard. You need a finishing-focused framework with an architectural design that scales down to getting started, then up as your application grows.

\n
\n
\n

Pyramid was made for just this. It\'s a Goldilocks Solution: not too small, not too big, just right.

\n

Pyramid The Start Small, Finish Big, Stay Finished Framework.

\n
\n
\n
\n
\n\n
\n

\n Start Small\n

\n \n

Getting started quickly and simply is a key attraction of lightweight frameworks. Equally, you get to choose what approaches to use for templating, database, security, and more, or use a convenient starting point with a scaffold. Pyramid excels at scaling down to the first hour of learning, while avoiding the pitfalls of framework magic.

\n
    \n
  • Start as a single-file module with little first-hour complexity
    • \n
  • Use a convenient scaffold to generate a sample project with your combination of subsystems
    • \n
  • Choose from a variety of templating, database, security solutions and more using the quality and convenience of Pyramid\'s add-on system
    • \n
  • Tap into a variety of high-quality documentation for evaluating, trying out, or doing advanced development with Pyramid
    • \n
  • Ask the growing Pyramid community for tips and successes
    • \n
\n
\n
\n

\n Finish Big\n

\n \n

Ambitious projects aspire to grow big without losing their dignity. Pyramid is uniquely equipped to scale with you. Its configuration, extension, and add-on system gives the skeleton to support your ambitions, and its architecture ensures that you don\'t lose those performance and quality features you started with.

\n
    \n
  • Include and configure Pyramid add-ons inside your application
    • \n
  • Override and customize core code and add-ons from the outside, without forking
    • \n
  • Build and deliver re-usable subsystems inside and outside our organization
    • \n
  • Less magic by forgoing globals and import side-effects
    • \n
  • Use the configuration system to keep your wiring separate from your code
    • \n
  • Numerous extension facilities built into the framework
    • \n
  • Use Pyramid as a "framework framework" to craft your own special-purpose, domain-specific web system
    • \n
  • Gain insight from our long-standing culture of systems that organically get big
    • \n
\n
\n

\n Stay Finished\n

\n \n

Pyramid\'s simple first hour helps you get started and its extensability helps you finish your ambitions. There\'s life after shipping. Pyramid helps keep your application finished by understanding the full life cycle of a professional web application.

\n
    \n
  • Deep commitment to API stability and bug fixing over the 120+ software releases
    • \n
  • Culture of 100% test and documentation coverage makes Pyramid a future-proof choice
    • \n
  • Keeping configuration out of code means less forking and side-effects
    • \n
  • Long history of repeatable deployments provides a community culture of helpful tips
    • \n
  • Top-notch performance even as Pyramid grows
    • \n
  • Deep extensibility and large-scale design patterns means you won\'t outgrow it
    • \n
  • Strong following of Python practices (WSGI, packages, virtual environments, first to support Python 3) means you won\'t be out of the Python mainstream
    • \n

\n Supports Your Decisions\n

\n \n

Full-stack frameworks provide built-in value by telling you what to do. But doing something different, or using something better, leads to the dreaded "fighting the framework". Pyramid starts from a very small base, providing many high-quality choices.

\n
    \n
  • Don\'t waste time fighting the framework\'s decisions
    • \n
  • "Only pay for what you eat" means less magic to live with in production
    • \n
  • No bundled templating system but instead, three very high-quality add-ons for Chameleon, Jinja2 and Mako
    • \n
  • Several form systems covering most of the common design tastes
    • \n
  • Choose your database, with add-on support for\n SQLAlchemy and others
    • \n

\n Function Decorators

\n

Pyramid is written in Python. All the features you know and love\n in the Python language, such as function decorators, are available\n to Pyramid developers. Here we show the function named\n home that returns a response. The function has a\n decorator @view_config which has a route assigned to\n it also named home.

\n
\n
\n
\n
\n
\n \n
\n 
\n            from wsgiref.simple_server import make_server\nfrom pyramid.config import Configurator\nfrom pyramid.response import Response\nfrom pyramid.view import view_config\n\n\n@view_config(\n    route_name=\'home\',\n    request_method=\'POST\'\n)\ndef home(request):\n    return Response(\'Welcome!\')\n\nif __name__ == \'__main__\':\n    with Configurator() as config:\n        config.add_route(\'home\', \'/\')\n        config.scan()\n        app = config.make_wsgi_app()\n    server = make_server(\'0.0.0.0\', 6543, app)\n    server.serve_forever()\n
\n
\n
\n

\n Predicates

\n

A test which returns True or False, and\n which narrows the set of circumstances in which views or routes may\n be called. Here we use predicates to limit matching of a view\n callable to a route name of home and to the\n POST HTTP request method.

\n
\n
\n
\n
\n
\n \n
\n 
\n            from wsgiref.simple_server import make_server\nfrom pyramid.config import Configurator\nfrom pyramid.view import view_config\n\n\n@view_config(\n    route_name=\'home\',\n    renderer=\'json\'\n)\ndef home(request):\n    return {"a": 1, "b": 2}\n\nif __name__ == \'__main__\':\n    with Configurator() as config:\n        config.add_route(\'home\', \'/\')\n        config.scan()\n        app = config.make_wsgi_app()\n    server = make_server(\'0.0.0.0\', 6543, app)\n    server.serve_forever()\n
\n
\n

\n Renderers

\n

Convert non-response return values that are later consumed by\n renderers. Using a renderer can make writing views that require\n templating or other serialization, like JSON, less\n tedious.\n

\n
\n
\n
\n
\n
\n \n
\n 
\n            """\n1. Run `env/bin/pip install pyramid_jinja2`\n2. Copy this template and put it in `templates/home.jinja2`:\n\n<!DOCTYPE html>\n<html lang="en">\n<head>\n    <title>{{ greet }}, {{ name }}</title>\n</head>\n<body>\n<h1>{{ greet }}, {{ name }}</h1>\n</body>\n</html>\n"""\nfrom wsgiref.simple_server import make_server\nfrom pyramid.config import Configurator\nfrom pyramid.view import view_config\n\n\n@view_config(\n    route_name=\'home\',\n    renderer=\'templates/home.jinja2\'\n)\ndef home(request):\n    return {"greet": \'Welcome\', "name": \'Akhenaten\'}\n\nif __name__ == \'__main__\':\n    with Configurator() as config:\n        config.include(\'pyramid_jinja2\')\n        config.add_route(\'home\', \'/\')\n        config.scan()\n        app = config.make_wsgi_app()\n    server = make_server(\'0.0.0.0\', 6543, app)\n    server.serve_forever()\n
\n
\n
\n

\n Asset Specifications

\n

Allows specifying the location of assets in a package. Here the\n asset is specified as a Jinja2 template home.jinja2,\n located in a subdirectory named templates. Within a\n package myapp, a colon delimits the package name from\n the location of the asset relative to the package, for example renderer=\'myapp:templates/home.jinja2\'.

\n
\n
\n

\n            from wsgiref.simple_server import make_server\nfrom pyramid.config import Configurator\nfrom pyramid.events import NewRequest\nfrom pyramid.events import NewResponse\nfrom pyramid.events import subscriber\n\n\n@subscriber(NewRequest, NewResponse)\ndef mysubscriber(event):\n    print(event)\n\nif __name__ == \'__main__\':\n    with Configurator() as config:\n        config.add_route(\'hello\', \'/\')\n        config.add_view(hello_world, route_name=\'hello\')\n        app = config.make_wsgi_app()\n    server = make_server(\'0.0.0.0\', 6543, app)\n    server.serve_forever()\n

\n Events and Subscribers

\n

An event is an object broadcast at interesting points during the\n lifetime of an application. A subscriber to an event allows you to\n run some code, such as resizing an uploaded image, sending email,\n or sending a message to a remote system. Here the decorated\n subscriber will be called for more than one event type,\n specifically for every new request and response objects.\n
